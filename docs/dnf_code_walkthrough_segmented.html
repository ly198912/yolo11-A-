这一段的作用

这段代码先把整个主程序需要的库全部导进来。cv2 用来处理图像，numpy 用来承接截图数据，pyautogui 用来截图，win32gui 和 win32con 用来找和摆放游戏窗口，Game 负责动作执行，MiniMapNavigator 负责小地图导航，Detector 负责 YOLO 检测。

执行逻辑

temp = pathlib.PosixPath 和后面的 pathlib.PosixPath = pathlib.WindowsPath 是一个兼容性小技巧，通常是为了防止某些模型或依赖在 Windows 下加载路径时，把路径对象当成了 Posix 风格。然后定义的几个窗口常量，是后面找 DNF 窗口和摆窗口位置时要用的“过滤条件”和“摆放目标位置”。

它在整套程序里的位置

这是总入口文件的“基础设施区”。还没开始真正刷图，但后面所有逻辑都会依赖这里导入的类和常量。你可以理解为程序开机后的准备工作。

片段 2：寻找 DNF 窗口
函数 _find_dnf_window()
def _find_dnf_window() -> int:
    candidates = []

    def _enum_windows(hwnd: int, _: object) -> None:
        if not win32gui.IsWindowVisible(hwnd):
            return

        title = win32gui.GetWindowText(hwnd).strip()
        class_name = win32gui.GetClassName(hwnd).strip()
        if not title and not class_name:
            return

        title_match = WINDOW_TITLE_KEYWORD in title
        class_match = class_name in WINDOW_CLASS_NAMES or WINDOW_TITLE_KEYWORD in class_name
        if not title_match and not class_match:
            return

        try:
            left, top, right, bottom = win32gui.GetWindowRect(hwnd)
            area = max(0, right - left) * max(0, bottom - top)
        except win32gui.error:
            area = 0

        candidates.append((area, hwnd, title, class_name))

    win32gui.EnumWindows(_enum_windows, None)
    if not candidates:
        raise RuntimeError("娌℃湁鎵惧埌 DNF 绐楀彛")

    candidates.sort(key=lambda item: item[0], reverse=True)
    hwnd = candidates[0][1]
    logger.info(
        "缁戝畾绐楀彛: hwnd={}, title={}, class={}",
        hwnd,
        candidates[0][2],
        candidates[0][3],
    )
    return hwnd

这一段的作用

这段代码的任务只有一个：在系统所有可见窗口里，把 DNF 窗口找出来，并返回它的句柄 hwnd。句柄你可以理解成“这个窗口在 Windows 系统里的身份证号”。

执行逻辑

程序先创建一个空列表 candidates，准备装所有疑似 DNF 的窗口。然后调用 win32gui.EnumWindows() 遍历系统里所有窗口，每遍历到一个窗口，就让内部函数 _enum_windows() 去检查。检查顺序是：先看它是否可见，再拿到窗口标题和类名；如果标题和类名都空，就跳过；如果标题和类名都不匹配 DNF 关键词，也跳过。只有疑似 DNF 的窗口才继续往下。接着它会读出窗口矩形大小，算一个面积，把面积、句柄、标题、类名一起存入列表。

它是干什么的

它的本质不是“后台绑定窗口”，而是“从系统里找出我要盯着哪一个游戏窗口”。最后代码按面积从大到小排序，选最大的那个候选窗口作为最终结果，这样可以避免同时开多个相关窗口时误选到很小的子窗口。

片段 3：把窗口摆到固定位置
函数 _place_window()
def _place_window(hwnd: int) -> None:
    win32gui.ShowWindow(hwnd, win32con.SW_RESTORE)
    left, top, right, bottom = win32gui.GetWindowRect(hwnd)
    width = max(1, right - left)
    height = max(1, bottom - top)
    win32gui.SetWindowPos(
        hwnd,
        win32con.HWND_TOPMOST,
        WINDOW_OFFSET_X,
        WINDOW_OFFSET_Y,
        width,
        height,
        win32con.SWP_SHOWWINDOW,
    )

这一段的作用

找到窗口之后，程序会把窗口恢复出来并移动到一个比较固定的位置，也就是屏幕左上附近的 (10, 10)。

执行逻辑

ShowWindow(..., SW_RESTORE) 的意思是如果窗口最小化了，就把它恢复。然后读取窗口当前的四个边界坐标，算出宽高。最后调用 SetWindowPos() 把窗口放到预设位置，并设置成置顶显示。

它是干什么的

这一步的目的，是让后面的截图区域更稳定。因为程序后面截图时依赖窗口客户区的位置，如果窗口乱跑、被挡住、缩到任务栏里，检测结果就会变得不稳定。

片段 4：计算客户区与截图
函数 _get_client_region() 与 _capture_client()
def _get_client_region(hwnd: int) -> Tuple[int, int, int, int]:
    left_top = win32gui.ClientToScreen(hwnd, (0, 0))
    client_rect = win32gui.GetClientRect(hwnd)
    left = int(left_top[0])
    top = int(left_top[1])
    width = int(client_rect[2] - client_rect[0])
    height = int(client_rect[3] - client_rect[1])
    if width <= 0 or height <= 0:
        raise RuntimeError("鑾峰彇娓告垙瀹㈡埛鍖哄け璐?)
    return left, top, width, height


def _capture_client(hwnd: int) -> Tuple[np.ndarray, Tuple[int, int]]:
    left, top, width, height = _get_client_region(hwnd)
    img = pyautogui.screenshot(region=(left, top, width, height))
    img_np = np.array(img)
    return img_np, (width, height)

这一段的作用

这段代码负责真正把游戏画面“拍下来”。注意它截的是窗口客户区，也就是游戏真正显示内容的区域，而不是包括标题栏和边框的整个窗口。

执行逻辑

ClientToScreen(hwnd, (0, 0)) 会把客户区左上角换算成屏幕坐标，GetClientRect() 会返回客户区的宽高。两个结果组合起来，就能得到截图需要的 left, top, width, height。然后 pyautogui.screenshot(region=...) 用这个矩形去截屏，截出来的是一个 PIL 图片，再转成 numpy 数组，后面才能送给 OpenCV 和 YOLO。

它是干什么的

这一段就是“给程序眼睛喂画面”。如果没有截图，后面的检测器、导航器、动作判断就全部无从谈起。

第二章：detector.py

这个文件负责什么： 用 YOLO 模型识别大画面里的玩家、怪物、Boss、门、金币、掉落物。它相当于主程序的第一只眼睛。

这一段的作用

这段代码负责把 YOLO 检测器真正准备好。它会设置图像大小、置信度阈值、IOU 阈值，然后加载本地模型文件 shzn.pt。

执行逻辑

img_size = 640 表示推理时把图像按 640 尺寸送进模型。conf_thres = 0.75 表示低于 0.75 置信度的目标就不要了。iou_thres = 0.45 用来做 NMS 去重。设备选择逻辑是：如果外面没强制指定设备，就自动判断显卡是否可用，可用就用 GPU 0，不可用就退到 CPU。之后用 YOLO(self.weights) 把模型载入内存，再从模型里拿到类别名称表 self.names，最后给每个类别随机生成一个画框颜色。

它是干什么的

这就是目标检测器的“开机步骤”。如果这一段没跑通，程序后面就根本不知道画面里哪里是怪、哪里是门、哪里是玩家。

片段 2：标签标准化与坐标格式转换
两个静态工具函数
    @staticmethod
    def _normalize_label(label: str) -> str:
        label = str(label).strip().lower()
        if "door" in label:
            return "door"
        return label

    @staticmethod
    def _xyxy_to_xywh_top_left(xyxy: np.ndarray) -> List[float]:
        x1, y1, x2, y2 = [float(v) for v in xyxy]
        return [x1, y1, max(0.0, x2 - x1), max(0.0, y2 - y1)]

这一段的作用

这一段不是做检测，而是负责把模型吐出来的数据整理成后面更容易用的格式。

执行逻辑

_normalize_label() 会把类别名统一转成小写字符串。如果类别名字里包含 door，就统统标准化成 door。这样就算模型里存在不同门标签，后面动作层也可以统一按“门”处理。_xyxy_to_xywh_top_left() 则把 YOLO 常用的矩形框格式 [x1, y1, x2, y2] 转成 [x, y, w, h]。这里的 x, y 是左上角，w, h 是宽高。

它是干什么的

本质上是在做“数据清洗和格式统一”。后面的 Game、导航器、门策略都依赖这种统一格式，否则每个模块都得自己重新解析框数据。

这一段的作用

这段代码是目标检测的真正入口。它接收一张截图，准备把它送进 YOLO 模型。

执行逻辑

第一步先判断输入图像是不是空，如果是空图，就直接返回空结果。这样可以避免程序某一帧截图失败时崩掉。然后它把图片从 RGB 转成 BGR。原因是前面的截图来自 pyautogui，通常是 RGB，而 OpenCV 和后面的一些显示逻辑习惯用 BGR。

它是干什么的

它相当于把外部输入的原始截图，转换成适合 YOLO 和 OpenCV 联合处理的图像格式，为真正的模型推理做准备。

这一段的作用

这段代码真正调用了模型，让模型对当前这一帧做目标检测。

执行逻辑

它是干什么的

从这一行开始，程序已经不是在“准备检测”，而是在“真正看图”。从这一步开始，程序会知道当前画面里到底有没有玩家、怪物、门和掉落物。

片段 5：读取检测框并整理输出
函数 detect() 的后半段

执行逻辑

这一段的最终成果有两样：一张可视化后的图，一份结构化对象列表。前者给人看，后者给程序做决策用。主刷图逻辑真正依赖的是这个 obj 列表。

第三章：door_strategy.py

这个文件负责什么： 当 A 星告诉程序“下一间房在右边”时，这个文件负责在当前画面检测到的多扇门里，挑出最适合往右走的那一扇。
片段 1：门的数据结构和方向反转

        "right": "left",

        "down": "up",
    }

这一段的作用

这一段先定义门候选的数据结构，再定义一个方向取反函数。门候选里面只存两样最关键的数据：包围框和中心点。

执行逻辑

DoorCandidate 是一个简单的数据类，后面导航器会把所有检测到的门包装成这种对象。opposite_direction() 则用来计算某个方向的反方向，比如往左的反方向是右，往上的反方向是下。

它是干什么的

这个文件后面的评分逻辑，需要知道上一轮的方向和它的反方向，好在挑门时减少“掉头回走”的概率。所以这里先把基础工具定义出来。

片段 2：计算门相对玩家的偏移
函数 _axis_offsets() 与 _direction_match()
def _axis_offsets(player: Point, door: Point) -> Tuple[float, float]:
    dx = door[0] - player[0]
    dy = door[1] - player[1]
    return dx, dy


def _direction_match(expected_direction: str, dx: float, dy: float, margin: float) -> bool:
    if expected_direction == "right":
        return dx > margin
    if expected_direction == "left":
        return dx < -margin
    if expected_direction == "up":
        return dy < -margin
    if expected_direction == "down":
        return dy > margin
    return True

这一段的作用

这段代码负责回答两个问题：第一，某扇门相对玩家偏在哪边；第二，这扇门算不算符合期望方向。

执行逻辑

_axis_offsets() 很简单，就是拿门中心减去玩家中心，得到 dx 和 dy。然后 _direction_match() 根据期望方向去判断。例如期望往右走时，只有 dx > margin 才算真正符合“右门”，也就是门必须明显在玩家右边，不是勉强靠右一点点。

它是干什么的

这里的 margin 就是一种容差。因为门的位置不可能永远刚好标准对齐，如果不留这个余量，就容易把斜方向或挨得很近的门误判成目标门。

片段 3：无门和无方向时怎么处理
函数 choose_best_door() 的前半段
def choose_best_door(
    doors: Sequence[DoorCandidate],
    player_center: Point,
    expected_direction: Optional[str],
    last_direction: Optional[str] = None,
    margin: float = 16.0,
) -> Optional[DoorCandidate]:
    if not doors:
        return None
    if expected_direction is None:
        return min(doors, key=lambda item: abs(item.center[0] - player_center[0]) + abs(item.center[1] - player_center[1]))

这一段的作用

这段是门选择函数的兜底起步逻辑。先处理一些简单情况，避免后面复杂评分没必要地执行。

执行逻辑

如果门列表为空，函数直接返回 None，表示当前没有可选门。如果有门但外部没有给出期望方向，那它就退而求其次，直接选离玩家最近的门。这里用的是曼哈顿距离形式的近似值，也就是 abs(dx) + abs(dy)。

它是干什么的

它保证了门策略在“没有路线提示”时也还能工作。也就是说，哪怕 A 星暂时没算出方向，程序也不会完全失去行动能力。

片段 4：核心评分循环
函数 choose_best_door() 的主循环
    reverse_direction = opposite_direction(last_direction)
    candidates: List[Tuple[Tuple[float, float, float], DoorCandidate]] = []
    fallback: List[Tuple[Tuple[float, float, float], DoorCandidate]] = []

    for door in doors:
        dx, dy = _axis_offsets(player_center, door.center)
        abs_dx = abs(dx)
        abs_dy = abs(dy)
        match = _direction_match(expected_direction, dx, dy, margin)

        if expected_direction in ("left", "right"):
            forward = dx if expected_direction == "right" else -dx
            sideways = abs_dy
        else:
            forward = dy if expected_direction == "down" else -dy
            sideways = abs_dx

        reverse_penalty = 0.0
        if reverse_direction and expected_direction == reverse_direction:
            reverse_penalty = 2000.0

        score = (-forward + reverse_penalty, sideways, abs_dx + abs_dy)
        if match:
            candidates.append((score, door))
        fallback.append((score, door))

这一段的作用

这是整份门策略里最关键的地方。它会给每一扇门打分，然后区分成“方向匹配候选”和“普通兜底候选”两类。

执行逻辑

程序先算出上一方向的反方向，作为后面反向惩罚的依据。然后对每扇门，计算门相对玩家的横纵偏移。match 表示这扇门是否满足期望方向，例如往右时门是否明显在右边。如果期望方向是左右，那么 forward 就看左右前进量，sideways 看上下偏移；如果期望方向是上下，则相反。之后它还会根据上一方向的反方向，加一个非常大的 reverse_penalty，用来减少程序刚刚往左，下一瞬间又马上掉头往右的情况。最终分数 score 是一个三元组：第一优先看前进值，第二优先看侧向偏移，第三优先看总距离。

它是干什么的

本质上这是一个启发式评分算法，不是深度学习。它的目标是把“房间级方向”翻译成“画面级门选择”，也就是把 A 星给出的抽象结果变成现实可走的那扇门。

片段 5：从候选中挑出最终的门
函数 choose_best_door() 的结尾

        candidates.sort(key=lambda item: item[0])
        return candidates[0][1]

    return fallback[0][1]

这一段的作用

这段代码负责从前面打好分的门里，真正选出最后那一扇。

执行逻辑

如果有“方向匹配”的候选门，就按分数排序，拿分数最优的那扇。如果一个都没有，说明当前画面里没有明显满足目标方向的门，那就退回到 fallback 列表里，从所有门里选最合理的一扇。

它是干什么的

这是门策略的最终决策出口。它保证了“优先尊重路线方向，但找不到完全匹配的门时，也要尽量选一个最像的门”，从而避免程序卡死在门选择这一步。

第四章：minimap_astar.py

这个文件负责什么： 它负责房间级路径规划。注意是“房间级”，不是“像素级”。它的输出不是角色脚下怎么一步一步拐弯，而是“下一间房应该往上、下、左、右哪边走”。
片段 1：节点结构与启发函数
文件顶部
import heapq
from dataclasses import dataclass
from typing import Iterable, List, Optional, Sequence, Tuple


GridPoint = Tuple[int, int]


@dataclass(order=True)
class _Node:
    f: int
    position: GridPoint
    g: int = 0
    h: int = 0
    parent: Optional["_Node"] = None


def heuristic(a: GridPoint, b: GridPoint) -> int:
    return abs(a[0] - b[0]) + abs(a[1] - b[1])

这一段的作用

这段定义了 A 星算法最基础的数据结构：节点，以及估价函数 heuristic()。

执行逻辑

_Node 里有五个核心字段：position 是当前格子坐标，g 是从起点走到这里的真实代价，h 是从这里到终点的估计代价，f 是总估价，也就是 g + h，parent 是当前节点的父节点，后面回溯路径时要用。heuristic() 用的是曼哈顿距离，因为当前地图只考虑上下左右四联通，不考虑斜走。

它是干什么的

这是 A 星算法的地基。没有节点结构，就没法维护路径搜索状态；没有启发函数，A 星也就退化成普通的广搜或 Dijkstra 风格搜索了。

片段 2：获取邻居节点
函数 _neighbors()
def _neighbors(position: GridPoint, grid: Sequence[Sequence[int]], priority: str) -> Iterable[GridPoint]:
    x, y = position
    directions = {
        "right": ((0, 1), (1, 0), (-1, 0), (0, -1)),
        "left": ((0, -1), (-1, 0), (1, 0), (0, 1)),
        "up": ((-1, 0), (0, -1), (0, 1), (1, 0)),
        "down": ((1, 0), (0, 1), (0, -1), (-1, 0)),
        "default": ((0, 1), (1, 0), (-1, 0), (0, -1)),
    }
    candidates = directions.get(priority, directions["default"])
    for dx, dy in candidates:
        nx = x + dx
        ny = y + dy
        if 0 <= nx < len(grid) and 0 <= ny < len(grid[0]) and grid[nx][ny] == 0:
            yield (nx, ny)

这一段的作用

这段代码负责从当前房间出发，找出能继续走到的上下左右邻居房间。

执行逻辑

程序先根据当前优先方向 priority，决定邻居的检查顺序。比如优先方向是 right，那就先看右，再看下、上、左。然后它逐个尝试这些偏移，计算新坐标 (nx, ny)。只有当新坐标没有越界，而且 grid[nx][ny] == 0 时，才把这个邻居返回出去。这里的 0 表示可通行，非 0 一般表示障碍。

它是干什么的

这一段既控制“能不能走”，也控制“同样可走时优先往哪边尝试”。所以它不只是列出邻居，还是 A 星路线偏好的一个调味器。

片段 3：A 星的初始化
函数 a_star() 的开头
def a_star(grid: Sequence[Sequence[int]], start: GridPoint, end: GridPoint, priority: str = "right") -> Optional[List[GridPoint]]:
    if start == end:
        return [start]

    open_list: List[_Node] = []
    start_node = _Node(f=0, position=start, g=0, h=heuristic(start, end), parent=None)
    start_node.f = start_node.g + start_node.h
    heapq.heappush(open_list, start_node)
    closed = set()
    best_g = {start: 0}

这一段的作用

这段代码负责把 A 星搜索的初始状态搭起来。

执行逻辑

如果起点和终点本来就是同一个房间，那直接返回只包含起点的路径。否则，创建开放列表 open_list，把起点包装成一个节点压进去。起点的真实代价 g 为 0，估计代价 h 由曼哈顿距离给出，总分 f = g + h。closed 集合用于记录已经确认处理过的节点，best_g 用来记录某个位置目前找到的最优真实代价，避免反复走更差的路径。

它是干什么的

这一步相当于在地图上摆下第一个探索点，并准备好之后要用的“已处理集合”和“最优代价表”。

片段 4：A 星主循环与回溯路径
函数 a_star() 的主体
    while open_list:
        current = heapq.heappop(open_list)
        if current.position in closed:
            continue
        closed.add(current.position)

        if current.position == end:
            path: List[GridPoint] = []
            node: Optional[_Node] = current
            while node is not None:
                path.append(node.position)
                node = node.parent
            return list(reversed(path))

        for next_pos in _neighbors(current.position, grid, priority):
            next_g = current.g + 1
            if next_g >= best_g.get(next_pos, 10 ** 9):
                continue
            best_g[next_pos] = next_g
            next_h = heuristic(next_pos, end)
            heapq.heappush(
                open_list,
                _Node(
                    f=next_g + next_h,
                    position=next_pos,
                    g=next_g,
                    h=next_h,
                    parent=current,
                ),
            )
    return None

这一段的作用

这一段就是 A 星算法真正跑起来的地方。它会持续从开放列表里拿出最优节点，不断向外扩展，直到找到终点。

执行逻辑

程序每次都从堆里弹出 f 值最小的节点作为当前节点。如果这个位置之前已经处理过，就跳过。否则把它加入 closed。如果当前节点就是终点，那么程序就沿着 parent 链一路往回走，把整条路径倒着拼出来，再反转成从起点到终点的顺序。如果不是终点，就去拿它的邻居。对于每个邻居，真实代价 next_g 就是当前 g + 1。如果这个代价不比之前记录的更优，就不要它；如果更优，就更新 best_g，重新计算估价 h，再把新节点压入堆。

它是干什么的

这就是整套房间级寻路的核心算法。它不会直接告诉角色在屏幕里怎么走，但会告诉程序从当前房间出发，最短路线应该经过哪些房间。

片段 5：把整条路径压缩成下一步方向
函数 judge_direction() 与 next_direction()
def judge_direction(starting_point: GridPoint, end_point: GridPoint) -> Optional[str]:
    x1, y1 = starting_point
    x2, y2 = end_point
    if x2 > x1:
        return "down"
    if x2 < x1:
        return "up"
    if y2 > y1:
        return "right"
    if y2 < y1:
        return "left"
    return None


def next_direction(grid: Sequence[Sequence[int]], start: GridPoint, end: GridPoint, priority: str = "right") -> Optional[str]:
    path = a_star(grid, start, end, priority=priority)
    if not path or len(path) < 2:
        return None
    return judge_direction(path[0], path[1])

这一段的作用

这段代码把 A 星整条房间路径，进一步压缩成当前最重要的那一个信息：下一步应该往哪边走。

执行逻辑

next_direction() 先调用前面的 a_star() 算出完整路径。如果路径为空，或者路径长度不足 2，说明当前无法继续前进，就返回空。否则只看路径的前两个房间：当前房间和下一个房间，然后交给 judge_direction() 去比较它们的行列变化，从而得出 up、down、left、right 之一。

它是干什么的

这是 A 星和整个刷图系统连接的关键出口。动作层不需要整条房间路径，它只需要知道“下一间房在哪个方向”，然后再去当前画面里找对应方向的门。

片段 5：按分辨率换算小地图裁剪区域

            int(round(y1 * scale_y)),
            int(round(x2 * scale_x)),

这一段负责从整张游戏截图中，把小地图准确裁出来。

执行逻辑

地图配置里存的裁剪区域是基于 1067×600 这类基准尺寸写的，但真实截图不一定总是这个大小。所以程序先拿当前截图的宽高，和基准宽高进行比例换算，计算出当前分辨率下的小地图区域。之后 extract_minimap() 根据换算出来的坐标直接把那块图裁剪出来。

它是干什么的

这一步本质上是在做分辨率适配。否则一旦游戏窗口大小略有变化，程序就会裁错位置，小地图识别就全乱了。

片段 6：模板匹配
函数 _match_template() 与 _match_template_debug()
    def _match_template(self, image: np.ndarray, template: np.ndarray, threshold: Optional[float] = None) -> List[Tuple[float, float]]:
        threshold = self.threshold if threshold is None else threshold
        image_gray = cv2.cvtColor(image, cv2.COLOR_BGR2GRAY)
        template_gray = cv2.cvtColor(template, cv2.COLOR_BGR2GRAY)
        result = cv2.matchTemplate(image_gray, template_gray, cv2.TM_CCOEFF_NORMED)
        ys, xs = np.where(result >= threshold)
        matches: List[Tuple[float, float]] = []
        h, w = template_gray.shape[:2]
        for y, x in zip(ys, xs):
            matches.append((x + w / 2.0, y + h / 2.0))
        return matches

    def _match_template_debug(self, image: np.ndarray, template: np.ndarray) -> Tuple[List[Tuple[float, float]], float]:
        image_gray = cv2.cvtColor(image, cv2.COLOR_BGR2GRAY)
        template_gray = cv2.cvtColor(template, cv2.COLOR_BGR2GRAY)
        result = cv2.matchTemplate(image_gray, template_gray, cv2.TM_CCOEFF_NORMED)
        _, max_val, _, _ = cv2.minMaxLoc(result)
        ys, xs = np.where(result >= self.threshold)
        matches: List[Tuple[float, float]] = []
        h, w = template_gray.shape[:2]
        for y, x in zip(ys, xs):
            matches.append((x + w / 2.0, y + h / 2.0))
        return matches, float(max_val)

这一段的作用

这是小地图识别的核心：模板匹配。程序拿一张小图标，在小地图上滑动比较，看哪里最像它。

执行逻辑

首先把大图和模板都转成灰度图，减少颜色差异带来的影响。然后调用 cv2.matchTemplate(..., TM_CCOEFF_NORMED) 做归一化相关性匹配，得到一张相似度热力图。程序会找出所有相似度大于阈值的位置，再把这些位置换算成模板中心点。调试版的函数则额外取出全局最大相似度 max_val，方便你判断某个模板当前到底匹配得好不好。

它是干什么的

这就是导航器的“第二只眼睛”。它不识别大画面里的怪和门，而是专门识别 UI 级别的小图标，比如玩家当前位置图标、Boss 图标、问号房图标。

片段 7：把像素位置换算成房间坐标
函数 compute_room_id()
    def compute_room_id(self, x: float, y: float, minimap_image: np.ndarray) -> Point:
        height, width = minimap_image.shape[:2]
        cell_w = width / self.spec.cols
        cell_h = height / self.spec.rows
        row = int(y // cell_h)
        col = int(x // cell_w)
        row = max(0, min(self.spec.rows - 1, row))
        col = max(0, min(self.spec.cols - 1, col))
        return row, col

这一段的作用

前面模板匹配只知道“图标在小地图里的哪个像素位置”，这一段负责把这个像素位置转换成“第几行第几列的房间”。

执行逻辑

程序先算出小地图每一个房间格子的宽和高。然后用图标中心点除以格子尺寸，得到它落在哪一行哪一列。最后再用 max 和 min 把结果夹住，防止因为边缘误差越界。

它是干什么的

这是把“图像空间”转换成“副本网格空间”的关键一步。A 星吃的不是像素坐标，而是房间坐标，所以必须经过这一层换算。

这一段的作用

这段代码把整张小地图上的重要房间全部识别出来，包括当前房间、Boss 房、问号房、精英房和下层入口房。

执行逻辑

首先从整张截图里裁出小地图。然后逐类模板去匹配。英雄图标用来确定当前房间，Boss 图标对应 Boss 房。问号房比较特殊，代码准备了多种不同样式的模板，只要匹配到其中一种就认作问号房。精英房先尝试常规模板，如果没有匹配到，就用特殊房模板兜底。下层入口则采用一个单独较低一点的阈值 0.62 去识别，因为这类图标可能更难匹配。最后把所有识别结果作为一个字典返回。

它是干什么的

这一步是在告诉程序：“我现在身处哪间房，值得去的目标房分别在哪里。”后面目标房的选择和 A 星的起终点，都靠这一段提供信息。

    def _door_candidates_from_objects(self, objects: Sequence[dict]) -> List[DoorCandidate]:
        doors: List[DoorCandidate] = []
        for item in objects:
            if "door" not in item:
                continue

                continue

_release_direction_keys() 会把上下左右四个方向键都松开。_release_cached_action() 更细一些，它先根据缓存动作字符串，比如 RIGHT_UP，把对应的键全部抬起，然后再调用一次总释放，最后把缓存动作清空。_distance() 则返回标准欧几里得距离，后面判断“门是否靠近入口”“角色是否突然跳到另一个位置”时都要用到。

它是干什么的

这类工具函数虽然不显眼，但非常关键。自动化程序最怕“键按住不放”或“前一帧的动作影响下一帧”，所以释放按键逻辑必须足够稳。

片段 4：推断进房方向与更新房间入口信息
函数 _infer_flow_direction_from_spawn() 与 _maybe_update_room_entry() 
    def _infer_flow_direction_from_spawn(self, player_center: Tuple[float, float]) -> Optional[str]:
        x, y = player_center
        edge_scores = {
            "LEFT": x,
            "RIGHT": self._width - x,
            "UP": y,
            "DOWN": self._height - y,
        }
        nearest_edge = min(edge_scores, key=edge_scores.get)
        opposite = {
            "LEFT": "RIGHT",
            "RIGHT": "LEFT",
            "UP": "DOWN",
            "DOWN": "UP",
        }
        if edge_scores[nearest_edge] > min(self._width, self._height) * 0.28:
            return None
        return opposite[nearest_edge]

    def _maybe_update_room_entry(self) -> None:
        if not self._player_xywh:
            return
        player_center = (self._player_xywh[0], self._player_xywh[1])
        previous = Game._last_player_center
        Game._last_player_center = player_center
        if previous is None:
            Game._room_entry_position = player_center
            Game._room_entry_time = time.time()
            Game._flow_direction = self._infer_flow_direction_from_spawn(player_center)
            return

        jump_distance = self._distance(player_center, previous)
        if jump_distance >= 150:
            Game._room_entry_position = player_center
            Game._room_entry_time = time.time()
            Game._flow_direction = self._infer_flow_direction_from_spawn(player_center)
            logger.info(
                "room transition inferred, entry={}, flow_direction={}",
                Game._room_entry_position,
                Game._flow_direction,
            )

这一段的作用

这段代码的任务是推断角色刚进入一个新房间时，是从哪个边缘出现的，从而反推出副本推进的大致流向。

执行逻辑

先看角色离屏幕四个边缘哪个最近。如果角色离左边最近，那通常意味着他是从左侧门进来的，因此接下来的推进方向大概率是往右。代码用一个 0.28 的比例阈值判断角色是否真的靠边，不靠边就不强行推断。_maybe_update_room_entry() 则会在每一帧更新玩家中心点，并对比上一帧位置。如果第一次记录玩家位置，就直接把当前位置记成入口；如果和上一帧距离突然跳了 150 像素以上，就认为可能刚刚换房，于是更新入口点、更新时间和推断出来的流向。

它是干什么的

这一步很重要，因为后面为了防止角色刚进房就回头走，会用到“入口保护”和“反向门过滤”。它的核心思想是：程序要记住自己是从哪里进来的。

片段 5：过滤回头门
函数 _filter_backtrack_doors()、_is_entry_protected()、_should_block_backtrack_direction() 
    def _filter_backtrack_doors(self, doors: Sequence[BoxDict], direction_hint: Optional[str]) -> List[BoxDict]:
        if not doors:
            return []

        if not Game._room_entry_position:
            return list(doors)

        now = time.time()
        protect_entry = now - Game._room_entry_time <= 3.0
        if not protect_entry:
            return list(doors)

        entry = Game._room_entry_position
        entry_side = self._reverse_direction(Game._flow_direction)
        filtered: List[BoxDict] = []
        for door in doors:
            x, y, w, h = door["xywh"]
            center = (x + w / 2.0, y + h / 2.0)
            near_entry = self._distance(center, entry) <= 240
            dx = center[0] - entry[0]
            dy = center[1] - entry[1]
            if abs(dx) >= abs(dy):
                door_side = "RIGHT" if dx > 0 else "LEFT"
            else:
                door_side = "DOWN" if dy > 0 else "UP"

            if not near_entry:
                filtered.append(door)
                continue

            if entry_side and door_side == entry_side:
                logger.info("ignore backtrack door at entry side {}: {}", entry_side, center)
                continue

            reverse_hint = self._reverse_direction(direction_hint)
            if reverse_hint and door_side == reverse_hint:
                logger.info("ignore reverse door for hint {}: {}", direction_hint, center)
                continue

            filtered.append(door)

        return filtered

    def _is_entry_protected(self) -> bool:
        return bool(Game._room_entry_position) and (time.time() - Game._room_entry_time <= 3.0)

    def _should_block_backtrack_direction(self, direction: Optional[str]) -> bool:
        if not direction or not self._is_entry_protected():
            return False
        entry_side = self._reverse_direction(Game._flow_direction)
        if not entry_side:
            return False
        return direction.upper() == entry_side

这一段的作用

这一段专门解决一个很常见的问题：角色刚进房，还没清怪，就因为门检测或路线判断抖动，立刻又跑回上一间房。

执行逻辑

过滤逻辑只在进房后的 3 秒保护期内生效。程序会拿每扇门和“入口位置”比较，先判断这扇门是不是离入口很近。只有离入口很近的门才会被重点检查。如果这扇门刚好处在“入口反方向的回头边”，就直接忽略；如果这扇门虽然不一定是入口门，但它正好与当前路线提示方向相反，也会被忽略。后面的 _should_block_backtrack_direction() 则是在没有明确门可选时，连后备方向都要做最后一道防回头检查。

它是干什么的

这是一套“防抖”和“防走回头路”的启发式规则。它不复杂，但对刷图稳定性特别重要，因为很多自动化程序就是死在频繁掉头和原地来回穿门上。

片段 7：选最近目标
函数 _get_nearest()
    def _get_nearest(self, cls_objs: Sequence[BoxDict], direct: Optional[str] = None) -> Optional[BoxDict]:
        nearest = None
        min_distance = float("inf")

        if not self._player_xywh:
            return None

        for item in cls_objs:
            xywh = list(item["xywh"])
            obj_x = xywh[0] + xywh[2] / 2
            obj_y = xywh[1] + xywh[3] / 2
            player_x = self._player_xywh[0]
            player_y = self._player_xywh[1]
            dx = obj_x - player_x
            dy = obj_y - player_y

            if direct:
                name = direct.lower()
                if name == "right" and dx <= 0:
                    continue
                if name == "left" and dx >= 0:
                    continue
                if name == "up" and dy >= 0:
                    continue
                if name == "down" and dy <= 0:
                    continue

            distance = (dx ** 2 + dy ** 2) ** 0.5
            if distance < min_distance:
                min_distance = distance
                nearest = {
                    "xywh": [obj_x, obj_y, xywh[2], xywh[3]],
                    "conf": item.get("conf", 0),
                }

        return nearest

这一段的作用

这一段负责在一堆同类目标里，比如一堆怪物或一堆门里，找出离玩家最近的那个。

执行逻辑

程序先确保自己知道玩家位置。然后对每个目标，把原始框换算成中心点，与玩家中心做差得到 dx、dy。如果传进来了方向限制 direct，就只保留该方向上的目标，比如期望往右走时只保留 dx > 0 的目标。之后用欧氏距离算当前目标离玩家多远，只要比目前最小距离还小，就更新为新的最近目标。

它是干什么的

战斗、拾取和过门都大量依赖这个函数。它体现的是一个很朴素但实用的策略：优先处理离自己最近、最容易立刻交互到的目标。

片段 8：把目标相对位置翻译成方向字符串
函数 _get_direction()
    def _get_direction(self, obj_box: Sequence[float]) -> Optional[str]:
        if not self._player_xywh:
            return None
        dx = obj_box[0] - self._player_xywh[0]
        dy = obj_box[1] - self._player_xywh[1]

        if dy < 0:
            if abs(dy) < 20:
                return "RIGHT" if dx > 0 else "LEFT"
            if self._player_xywh[1] - obj_box[1] < abs(dx):
                return "RIGHT_UP" if dx > 0 else "LEFT_UP"
            return "UP"
        if dy > 0:
            if abs(dy) < 20:
                return "RIGHT" if dx > 0 else "LEFT"
            if dy < abs(dx):
                return "RIGHT_DOWN" if dx > 0 else "LEFT_DOWN"
            return "DOWN"
        return None

这一段的作用

这段代码负责把“目标在玩家的哪边”翻译成程序能执行的方向字符串，比如 LEFT、UP、RIGHT_DOWN。

执行逻辑

程序先计算目标中心相对玩家中心的横纵偏移。如果目标在玩家上方，先看上下差是否很小，若很小就当成左右方向；否则再比较上下偏差和左右偏差，决定是“偏斜上方”还是“纯上方”。目标在下方时也是同样逻辑。这里的 20 像素阈值就是一种容差，允许目标虽然略高一点点，但仍然被当作左右方向处理。

它是干什么的

这是动作层把空间关系变成按键方向的翻译器。后面的打怪、拾取、朝门移动，几乎都是先调用它，再把结果扔进按键函数。

片段 10：判断角色是否卡住
函数 _update_motion_state()
    def _update_motion_state(self, expect_motion: bool, player_visible: bool) -> bool:
        if not self._player_xywh:
            return False
        if not expect_motion or not player_visible:
            Game._position_history.clear()
            Game._stuck_count = 0
            return False

        current = (self._player_xywh[0], self._player_xywh[1])
        Game._position_history.append(current)
        if len(Game._position_history) > 8:
            Game._position_history.pop(0)

        if len(Game._position_history) < 6:
            return False

        xs = [point[0] for point in Game._position_history]
        ys = [point[1] for point in Game._position_history]
        spread_x = max(xs) - min(xs)
        spread_y = max(ys) - min(ys)
        if spread_x < 18 and spread_y < 18:
            Game._stuck_count += 1
        else:
            Game._stuck_count = 0

        return Game._stuck_count >= 2

这一段的作用

这段代码专门判断：“程序明明在尝试移动，但角色是不是其实几乎没动？”如果是，就把它视为卡住。

执行逻辑

只有在两个条件都满足时才会做卡住检测：第一，程序当前确实在期待角色运动；第二，玩家目标本身确实被检测到了。然后它会记录最近几帧玩家中心点位置，最多保留 8 个。如果历史点不够 6 个，先不下结论。等样本够了以后，程序计算这些点在 X 和 Y 方向的扩散范围。若横向和纵向扩散都小于 18 像素，就认为角色基本没动，卡住计数加 1；否则就清零。连续两次满足这个条件，才正式判断为卡住。

它是干什么的

这是一套很典型的“位置历史容差法”。它不要求程序精准知道碰撞发生了什么，只看结果：最近一段时间位置是不是几乎没变。

片段 11：卡住恢复与丢失玩家恢复

    def _recover_from_stuck(self, reason: str) -> None:
        now = time.time()
        if now - Game._last_recover_time < 1.2:
            return

        try:
            Game._last_recover_time = now
            Game._stuck_count = 0
            Game._position_history.clear()
            logger.warning("stuck recovery triggered: {}", reason)
            current_hint = Game._action_cache or self._current_door_direction() or Game._flow_direction or "RIGHT"
            self._release_cached_action()
            reverse_hint = self._reverse_direction(current_hint) or "LEFT"
            pattern = Game._recover_step % 4
            Game._recover_step += 1

            if pattern == 0:
                self._tap_direction(reverse_hint, 0.18)
                self._tap_direction("DOWN", 0.12)
            elif pattern == 1:
                self._tap_direction(reverse_hint, 0.18)
                self._tap_direction("UP", 0.12)
            elif pattern == 2:
                for step in ("RIGHT", "DOWN", "LEFT", "UP"):
                    self._tap_direction(step, 0.1)
            else:
                self._tap_direction(reverse_hint, 0.22)
                self._tap_direction(current_hint if isinstance(current_hint, str) else "RIGHT", 0.1)

            Game._action_cache = None
        except Exception as exc:
            logger.exception("stuck recovery failed: {}", exc)
            self._release_cached_action()
            Game._action_cache = None

    def _recover_missing_player(self) -> None:
        if Game._player_missing_count < Game._missing_player_recover_threshold:
            return
        logger.warning("player lost for {} frames, try recovery", Game._player_missing_count)
        self._recover_from_stuck("player_missing")
        Game._player_missing_count = 0

这一段的作用

一旦程序判断自己卡住，或者连续很多帧都找不到玩家，它就会进入恢复逻辑，尝试把角色重新从僵局里拽出来。

执行逻辑

首先有一个恢复冷却，避免短时间反复乱按。然后清空卡住计数和位置历史，并选择一个“当前提示方向”。这个提示方向的来源优先级是：当前缓存动作，其次是路线方向，再其次是房间流向，实在不行默认往右。然后计算反方向，按四种模式轮流尝试脱困：反方向再下、反方向再上、绕一个小矩形、反方向再朝当前方向试探。_recover_missing_player() 则在连续很多帧都丢失玩家检测结果时，直接复用这套卡住恢复逻辑。

它是干什么的

这是动作层的“自救系统”。因为自动化里总会遇到碰撞、门槛、怪物挡路、识别抖动等问题，如果没有恢复策略，程序很容易永远卡在某个角落。

        amplitude = 20.0

        if phase == 0:
            command = "DOWN"
            if current_y - start_y >= amplitude:
                Game._route_search_phase = 1
                Game._route_search_start = current
                command = "RIGHT"
        elif phase == 1:
            command = "RIGHT"
            if current_x - start_x >= amplitude:
                Game._route_search_phase = 2
                Game._route_search_start = current
                command = "UP"
        elif phase == 2:
            command = "UP"
            if start_y - current_y >= amplitude:
                Game._route_search_phase = 3
                Game._route_search_start = current
                command = "RIGHT"
        else:
            command = "RIGHT"
            if current_x - start_x >= amplitude:
                Game._route_search_phase = 0
                Game._route_search_start = current
                command = "DOWN"

        logger.info(
            "up fallback sweep: phase={}, start={}, current={}, command={}",
            Game._route_search_phase,
            Game._route_search_start,
            current,
            command,
        )
        Game._action_cache = self._move(command, is_slow=True, _action_cache=Game._action_cache)

这一段的作用

这段代码是一个很有意思的特例：当后备路线提示要求向上，但当前又没检测到清晰门目标时，程序不会傻傻一直往上顶，而是执行一套扫图式搜索动作。

执行逻辑

程序把这个搜索过程看成一个四阶段小状态机。每一阶段都会朝一个固定方向慢慢走，直到位移超过 amplitude = 20 像素，就切到下一个阶段。整体轨迹大致是“先向下一点，再向右，再向上，再向右”，形成一种小范围扫描动作。这样做是为了在没有明显上门时，尝试移动到更容易看到或进入上门的位置。

它是干什么的

这不是严格意义上的寻路算法，而是一种启发式搜门手法。它反映了实际刷图中的经验：某些门尤其是上门，角色必须站到特定位置才能顺利接近，所以程序会主动扫一下位置。

片段 15：通用后备路线移动与位置象限判断
函数 _fallback_route_move() 与 _check_position()
    def _fallback_route_move(self, direction: str) -> None:
        normalized = direction.upper()
        if normalized == "DOWN":
            self._reset_route_search()
            if not self._player_xywh:
                Game._action_cache = self._move("DOWN", is_slow=True, _action_cache=Game._action_cache)
                return
            player_x = self._player_xywh[0]
            if player_x > self._width * 0.62:
                normalized = "LEFT_DOWN"
            elif player_x < self._width * 0.38:
                normalized = "RIGHT_DOWN"
        elif normalized == "UP":
            self._fallback_up_search_move()
            return
        else:
            self._reset_route_search()

        Game._action_cache = self._move(normalized, is_slow=True, _action_cache=Game._action_cache)

    def _check_position(self) -> str:
        if not self._player_xywh:
            return "RIGHT_DOWN"
        player_x = self._player_xywh[0]
        player_y = self._player_xywh[1]

        if player_x < self._width / 2:
            return "LEFT_UP" if player_y < self._height / 2 else "LEFT_DOWN"
        return "RIGHT_UP" if player_y < self._height / 2 else "RIGHT_DOWN"

这一段的作用

当前画面里如果没有明确目标，比如怪、物品、门都不明显，但导航器又给了一个路线方向，这段代码负责根据这个方向做一个兜底移动。

执行逻辑

如果后备方向是下，程序会根据玩家在屏幕横向位置，适当改成左下或右下，这样更容易撞到真正的下门。若方向是上，则交给前面的“向上扫门模式”。如果方向是左右，则直接慢速朝该方向移动。_check_position() 则根据玩家中心落在屏幕哪个象限，返回一个组合方向字符串，后面某些多步方向列表逻辑会用它来判断当前是否已经偏到某个区域。

它是干什么的

这是没有明确目标时的“走位指南”。它并不聪明，但足够实用：至少保证程序在没有门框或门识别不稳时，不会完全原地发呆。
